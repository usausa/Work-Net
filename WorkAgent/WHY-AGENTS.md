# なぜ「チャット」ではなく「エージェント」なのか / なぜ Agent Framework のような層が必要か

> 対象: Microsoft Agent Framework **1.0 GA**（本リポジトリ参照は 1.10.0）/ .NET 10
> 作成日: **2026-06-20**（記述は同日時点の確認に基づく。公式 Overview の更新日: 2026-06-11）
> 出典: 末尾の[参考](#参考一次情報2026-06-20-確認)に一次情報リンクを掲載。

「LLM を呼ぶだけ（チャット）」と「エージェント」は何が違い、いつエージェントが要るのか。
さらに、なぜ素のクライアントではなく **Agent Framework のような層** を挟むのか——をまとめます。
機能の一覧は [FEATURES.md](FEATURES.md)、動くサンプルは [README.md](README.md) を参照してください。

---

## 🤔 1. チャットとエージェントの違い

**チャット（単発の LLM 呼び出し）** は「メッセージを渡すと文章を返す」だけ。外部の事実取得も、行動も、状態の記憶もしません。
**エージェント** は LLM を**ループ**の中で使います——モデルが「次に何をするか」を判断し、ツールを呼び、結果を観測し、目的を達成するまで繰り返します（＝**エージェントループ**）。

| 観点 | 💬 チャット（単発 LLM） | 🤖 エージェント |
| --- | --- | --- |
| 外部の事実・行動 | 単発呼び出しでは不可（テキストを返すのみ） | ツール / MCP を呼んで取得・実行できる |
| 多段の手順 | 1 回の応答で完結 | 自分で計画し複数ステップを回す |
| 状態 | ステートレス（毎回ゼロから） | セッションで履歴・文脈を保持 |
| 主導権 | 人間が逐一指示 | 目的を与えると自律的に進める |
| 例 | 「この文を要約して」 | 「未処理の問い合わせを調べて返信案を作って」 |

> 📌 エージェント＝**LLM＋ツール＋状態＋ループ**。「考える」だけのチャットに「調べる・実行する・覚える」を足したもの。

> 🔌 **「MCP やツールを使えばチャットでも外部とつながるのでは?」** — はい、そのとおりで、
> **ツール呼び出し / MCP こそがチャットをエージェントに変える境界**です。単発の LLM 呼び出しは
> 「ツールを呼びたい」という*要求*をテキストで返すだけで、実際に**実行し結果を取り込み、必要なら繰り返す**
> 主体が別に要ります。この**ループを回す主体がエージェント**です。
> MCP は外部ツール/データを**標準プロトコルで供給する**仕組み（ツールの提供側を差し替え可能にする）であり、
> ループ・状態・制御・観測といった**エージェント側の責務をなくすものではありません**。むしろそれらを担うのが
> Agent Framework のような層です。公式 Overview もツール・MCP サーバーの呼び出しを**エージェントの機能**として説明しています。

## 🎯 2. なぜエージェントが必要か

チャットだけでは届かない、次のような要求がエージェントを必要にします。

- **🔧 外部とつながる必要**: 最新情報・社内データの取得や、メール送信・チケット作成などの**行動**。モデル単体にはできません。
- **🧩 多段タスクの自律遂行**: 「調べて→判断して→実行して→確認する」を、人が毎手順を指示せずに回したい。
- **🧠 文脈の保持**: 複数ターンにわたる会話・作業で、過去のやり取りや中間結果を覚えていてほしい。
- **🔁 結果に応じた分岐**: ツールの結果を見て次の行動を変える（失敗ならリトライ、不足なら追加調査）。

これらは「1 回の応答」では表現できず、**モデルが行動を選び結果を取り込むループ**＝エージェントが要ります。

## 🧭 3. 向くケース / 向かないケース（過剰適用に注意）

エージェントは万能ではありません。**単純な処理にエージェントを使うのは過剰**で、コスト・遅延・不確実性を増やします。
公式も「タスクを関数で書けるなら、エージェントではなく関数を使え」と明言しています。

| ✅ エージェントが向く | ⚠️ エージェントは不要（過剰） |
| --- | --- |
| 目的が開放的・対話的 | 入出力が決まった定型処理 |
| 自律的なツール利用・計画が要る | 単発の LLM 呼び出しで足りる |
| 手順や分岐が事前に決めきれない | **関数で書けるならその方が速く・確実** |

さらに、**手順と分岐が事前に決まっている**業務プロセスは、自由なエージェントより
**ワークフロー**（グラフ型・型安全・チェックポイント・人手承認）が適します（公式の整理）。

| エージェントを使う | ワークフローを使う |
| --- | --- |
| タスクが開放的・対話的 | 手順が明確に定義済み |
| 自律的なツール利用・計画が必要 | 実行順を明示的に制御したい |
| 単一の LLM 呼び出し（＋ツール）で足りる | 複数エージェント/関数の協調が必要 |

> 🧱 **判断のめやす**: 関数で書ける → **関数**。1 回の補完で足りる → **素の ChatClient**。
> 自律ループ・ツール・状態が要る → **エージェント**。手順固定・分岐・監査が要る → **ワークフロー**。

## 💼 4. ユースケース

エージェントが価値を出す代表例と、そこで効く機能（詳細は [FEATURES.md](FEATURES.md)）。

| ユースケース | チャットでは何が不足するか | 主に使う機能 |
| --- | --- | --- |
| カスタマーサポート自動化 | 注文照会・返金など**実データ操作**＋多ターン文脈 | ツール / セッション / ツール承認 |
| コーディング支援 | ファイル読み書き・**計画と段階実行** | ファイルアクセス / TODO / モード / 承認 |
| 社内ドキュメント QA | 学習外の**最新/社内情報**と**出典**提示 | RAG（テキスト検索） |
| リサーチ・調査 | タスク分解・複数ソース収集・長時間処理 | TODO / マルチエージェント / バックグラウンド |
| 業務プロセス自動化（請求処理など） | **手順固定・分岐・監査・耐障害性** | ワークフロー（＋承認） |
| 運用・SRE 補助 | メトリクス取得→是正アクション→**人手承認** | ツール / 承認 / テレメトリ |
| 構造化データ抽出（バッチ） | 後続処理に渡す**型付き出力** | 構造化出力（※単発でも可・エージェント必須ではない） |

> 最後の例のように、**ツール不要・1 回で完結**するなら、エージェント化せず単発呼び出しで十分なこともあります（過剰適用の回避）。

## 🧩 5. なぜ汎用エージェントではなく専用エージェントか

「MCP やツールで何でも繋げられるなら、巨大な"何でも屋"エージェント 1 体でよいのでは?」——
実際には、**役割を絞った専用エージェントを複数**組み合わせる方が、精度・安全・コスト・保守のどれでも有利です。

| 観点 | 汎用（何でも屋）1 体だと | 専用エージェントだと |
| --- | --- | --- |
| 精度・ツール選択 | ツールが多すぎて選択を誤る/迷う、指示も薄まる | 関連ツールだけ → 選択が正確 |
| コスト・遅延 | 全ツールのスキーマが毎回文脈に乗り肥大 | 必要分のみ → トークン/遅延を削減 |
| セキュリティ | 強力な権限を 1 体に集中（爆発半径が大きい） | 最小権限・承認をスコープ化 |
| 保守・評価 | 巨大プロンプトでテスト困難、変更が全体に波及 | 小さく分割 → テスト/評価/個別更新が容易 |
| モデル選択 | 1 モデルに固定 | タスク毎に最適モデル（安い/速い）を割当 |
| 再利用・構成 | 密結合で使い回しにくい | 部品として組み合わせ可能 |

専用エージェントは、**エージェントのツール化**（[FEATURES.md](FEATURES.md) の E1／サンプル [07](Feature07MultiAgent/README.md)）や
**ワークフロー**（E2）で束ねます。調整役が、ハードウェア担当・ストレージ担当などの専門エージェントへ委譲して統合する、といった構成です。

> 🧭 **2 つの軸で整理する**（混同しやすい別々の判断です）。
> - **軸1: 対話的・開放的か / 手順が固定か** → 「エージェント」か「ワークフロー」か
> - **軸2: 役割が広い（汎用）か / 専門性が高い〔狭く深い〕か** → 「汎用 1 体」か「専用」か
>
> よって **「対話的 ＋ 専門性が高い」→ 専用エージェントが適切**です。
> ただし専門性が *1 つ* なら「専用＝よく絞った **1 体**」で十分。**複数の専門性を協調**させる必要が出た段階で、
> 専門エージェントへ**分割して連携**させます（ツール化／ワークフロー）。

> ⚖️ **ただし過剰分割も禁物**（モノリス vs マイクロサービスと同じ）。ツールが少なく権限も限定的で、
> 探索・小規模の段階なら、1 体の汎用エージェントで十分です。**規模・信頼性・安全・コスト**が効いてきた段階で専用化・分割します。

## 🏗️ 6. なぜ Agent Framework のような「層」が必要か

エージェントを素のクライアントだけで作ると、**本質でない定型処理を大量に自前実装**することになります。
Agent Framework は、その共通基盤（モデルクライアント、状態管理のセッション、メモリの文脈プロバイダー、
行動を横取りするミドルウェア、ツール連携の MCP クライアントなど）を提供します（公式 Overview）。

### 自前実装 vs フレームワーク

| 関心事 | 素のクライアントで自前実装すると | Agent Framework では |
| --- | --- | --- |
| ツール実行ループ | 呼び出し要求の検出→実行→差戻し→再呼出を手で回す | メソッドを渡すだけで内蔵 |
| 会話状態・永続化 | 履歴を自前で保持・直列化・復元 | セッションが標準化 |
| 文脈・メモリ注入 | 毎回プロンプトを手組み | コンテキストプロバイダー |
| 横断処理（ログ/トレース/リトライ） | 各所に散らばる | ミドルウェアで合成 |
| 人手承認（HITL） | 中断・再開を自作 | 承認の中断点を標準提供 |
| マルチエージェント連携 | 呼び出し規約を自作 | エージェントのツール化 / ワークフロー |
| プロバイダー差し替え | SDK ごとに書き分け | 共通の `AIAgent` 抽象に集約 |
| 可観測性・評価 | 計測・採点を自作 | OpenTelemetry / 評価器 |

> 要するにフレームワークは、**「LLM を賢く使うための配管」を肩代わり**し、開発者が
> ドメインのロジック（どんなツール・どんな指示・どんなフロー）に集中できるようにします。

### 公式の位置づけ

Agent Framework は **AutoGen の単純なエージェント抽象** と **Semantic Kernel のエンタープライズ機能**
（セッション状態・型安全・ミドルウェア・テレメトリ）を統合し、さらに**グラフ型ワークフロー**を加えた、
両者の後継です（公式 Overview「Why Agent Framework?」）。標準化により、**試作から本番運用まで同じ土台**で進められます。

## ⚖️ 7. まとめ（導入判断のめやす）

```
タスクを関数で書ける？        ── はい ─→ 関数で実装（AI 不要）
   │ いいえ
1 回の LLM 呼び出しで足りる？  ── はい ─→ 素の ChatClient（チャット）
   │ いいえ（ツール・状態・自律ループが要る）
手順・分岐が事前に決まっている？─ はい ─→ ワークフロー（型安全・監査・耐障害性）
   │ いいえ（開放的・対話的）
   └─→ エージェント ──→ 役割が広い/権限が強い/規模が大きい？─ はい ─→ 専用エージェントに分割し連携
                              （必要な機能は FEATURES.md から選ぶ）
```

- **過剰適用を避ける**: 単純な処理にエージェントは不要。コスト・遅延・不確実性が増えるだけ。
- **専用化で精度と安全を上げる**: 役割・ツール・権限を絞った専用エージェントを、ツール化やワークフローで束ねる（過剰分割は避ける）。
- **本番要件はフレームワークに委ねる**: 状態管理・承認・トレース・評価などは自作せず標準機能を使う。
